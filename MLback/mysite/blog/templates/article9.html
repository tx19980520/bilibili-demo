Git Workflow

What is Git Workflow​

A Git Workflow is a recipe or recommendation for how to use Git to accomplish work in a consistent and productive manner.

Given Git's focus on flexibility, there is no standardized process on how to interact with Git. Every group has different situations so We shouldn't ponderously use the workflows given below. Some things to consider when evaluating a Git workflow are:

  • Does this workflow scale with team size?
  • Is it easy to undo mistakes and errors with this workflow?
  • Does this workflow impose any new unnecessary cognitive overhead to the team?

Centralized Workflow

It is very friendly for groups transitioning from SVN, which can simply consider the common server as a codebase. Every developer can get files from the server and submit the new to the server.

How Centralized Works

  1. The team should Initialize the central repository(Maybe is empty or you can import an existing Git or SVN repository).

  2. Then developers begin with cloning the central repository. They edit files and submit the changes. These new submits are stored locally.Developers can repeat this process as many times as they want without worrying about what’s going on in the central repository. The local repository is private.

  3. If one wants to change the central repository, he/she can use standard Git comment progress and these changes will need to be pushed to share with other developers at the project.

  4. About managing conflicts, for example, if now you want to submit your changes and others have changed the central repository ,the git will refuse your submit and you should get the latest files and change something basing on it.(If the git allow you to submit, others' work will not exist!)

Feature Branching

Feature Branching is a logical extension of Centralized Workflow. The core idea behind the Feature Branch Workflow is that all feature development should take place in a dedicated branch instead of the master branch.

It also means the master branch should never contain broken code, which is a huge advantage for continuous integration environments. Also it makes the pull requests come true,which means After one branch has completed and submitted, instead of immediately mingling with the master, It will pull requests.Before the central repository take in the branch,the developer can review it at any time.

### Git Workflow

In my opinion, we can use recursion to look at the whole workflow, which is an enhanced Feature Branching.

Firstly, gitflow uses two "master"(Actually one is "develop",but it is important as the master)branches to record the develop history. The master saves the official release history and the develop is integrating various functions.

You can regard every branch as a master, which means every branch can have branch.This workflow doesn't add any new concepts or commands beyond what’s required for the Feature Branch Workflow. Instead, it assigns very specific roles to different branches and defines how and when they should interact.

How Git Workflow Works

  1. After the team creates the central repository, it has the master and the develop, The master can not be changed unless the official release of the new edition or fix bugs and release again.(the hotfix branch, which is the branch of the master)
  2. Before releasing, the develop is integrating new functions and review codes.(If we have hotfix branch we should mingle it with the development immediately)
  3. every function has a new branch and the function will be the branch of the develop.

Forking Workflow

Forking workflow is fundamentally different from others. Instead of one codebase, it has one unique formal codebase and plenty of server-side codebases for every contributer.

How Forking Workflow Works

  1. The official creates an official codebase and publishes a copy in the server.

  2. Contributers can clone the code in the server and create server-side codebases for their work and submit the new to it. Also they will ask the administrator to add their codes into the official codebase.

  3. the administrator will decide to choose whether your contribution is added into the official database.

Why we choose Git Workflow

In my opinion, it has several merits:

  1. Revision control

Revision control is also the merit of the SVN, but it is also important.

  1. Make pull requests come true

We emphasize the pull requests hereinabove, which is very convenient for contributers to contribute their codes and prevent plenty of bugs surge into the master and make all collapse.

  1. Distributed system

Git Workflows make it decentralized and every developer can have a local repository. Developers can review their code at local and make the official repository relatively safe.
